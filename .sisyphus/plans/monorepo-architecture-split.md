# Monorepo Architecture Split

## TL;DR
> **Summary**: Restructure Specra from a single `src/` tree into a Bun workspace + Turbo monorepo with explicit browser-safe protocol, agent runtime, server adapter, and web app boundaries. The migration removes duplicated session/protocol types, eliminates Web → runtime imports, preserves familiar dev commands, and makes `bun run build` produce a standalone Bun binary with embedded Vite Web assets.
> **Deliverables**:
> - `apps/server` Hono/Bun host and binary entry point.
> - `apps/web` React/Vite frontend.
> - `packages/protocol` browser-safe protocol types, session projection types, and pure `reduceStreamEvent()`.
> - `packages/agent-core` runtime package containing agents, tools, store implementation, config/provider, MCP, LSP, memory, commands, compact, background hooks, prompt, projects, workflow, security, utilities, and `createSpecraRuntime()`.
> - Turbo + Bun workspace scripts preserving root `server`, `web`, `dev`, `build`, `typecheck`, and `test`; remove `start` and `server:prod`.
> - True single-binary build: Vite build → generated `with { type: "file" }` web asset manifest → `bun build --compile` → `dist/specra`.
> - Root architecture boundary tests enforcing package dependency rules.
> **Effort**: XL
> **Parallel**: YES - 5 waves, with early audit/protocol/runtime tasks mostly sequential and later package/test/build tasks partially parallelizable.
> **Critical Path**: Task 1 dependency audit → Task 2 workspace scaffold → Task 3 protocol extraction → Task 4 runtime store split → Task 6 runtime facade split → Task 8 server move → Task 9 web move → Task 11 binary embedding → Task 13 architecture tests → Final Verification Wave.

## Context

### Original Request
User requested a careful architecture plan in Chinese because current code structure mixes frontend, backend, and agent modules in one repository and duplicates many type definitions. The user proposed a Turbo monorepo with at least Agent, Docs, Server, and Web modules, asked to research competitors from `docs/links.md`, and explicitly required sufficient research and discussion before planning.

### Interview Summary
- User accepted a monorepo migration, dependency cleanup, type duplication removal, and Bun standalone binary packaging as required scope.
- User rejected far-future SDK/plugin/platform planning for this migration.
- User rejected a standalone `packages/tools` split for now; tools stay inside agent runtime.
- User accepted `@specra/agent-core` as the runtime package and `@specra/protocol` as the browser-safe protocol package.
- User accepted dependency direction:
  ```text
  apps/web -> @specra/protocol
  @specra/agent-core -> @specra/protocol
  apps/server -> @specra/agent-core
  ```
- User explicitly accepted forbidding `apps/web -> @specra/agent-core`, even though both are workspaces under the same monorepo.
- User decided docs stay as root `docs/` for this migration; no `apps/docs` yet.
- User decided production does not need `bun run start`; root `start` and `server:prod` should be removed. Production means executing the compiled binary generated by `bun run build`.
- User required existing dev commands to remain: root `bun run server`, `bun run web`, `bun run dev`, `bun run build`, `bun run typecheck`, and `bun test`.
- User required that if implementation reveals any new serious problem, package-boundary ambiguity, circular dependency, binary blocker, or shared/public API decision, the executor must research thoroughly and discuss with the user instead of deciding silently.

### Metis Review (gaps addressed)
- `reduceStreamEvent` is a runtime function imported by Web, not only a type. Decision: extract a small browser-safe pure protocol reducer into `@specra/protocol`; do not share Zustand store, persistence, or runtime store implementation.
- `tools/` → `agents/` has a back-edge through `AgentFactoryLike`. Decision: keep inside `@specra/agent-core`, move/downscope the minimal delegation interface to a neutral internal boundary such as `packages/agent-core/src/delegation/types.ts`; do not place it in protocol.
- Workflow domain is imported by tools/projects/server. Decision: keep workflow under `@specra/agent-core`; do not create `@specra/workflow`.
- `src/main.ts` mixes runtime creation and server boot. Decision: move `SpecraRuntime` and `createSpecraRuntime()` to `packages/agent-core/src/runtime.ts`; `apps/server/src/main.ts` becomes the Hono/Bun host entry.
- `PermissionService`, `AskUserService`, and current `AgentRunner` are HTTP/SSE deferred-interaction and job-orchestration services. Decision: keep them in `apps/server`.
- `AgentRunner` currently uses `instanceof ConfiguredAgent`. Decision: remove the server’s concrete class dependency by adding a server-facing `dispatchCommand(...)` facade on `SpecraRuntime`; server calls the facade and never imports `ConfiguredAgent`.
- Server access to agent-core: use package-root public exports and `SpecraRuntime` facade only; forbid deep imports into agent-core internals from `apps/server`.
- MCP, LSP, memory, projects, workflow, commands, compact, background, prompt, security, and utilities stay in `@specra/agent-core` for this migration.
- Zod policy: `@specra/protocol` may depend on `zod` only for browser-safe wire/protocol schemas and `.strict()` validation objects; protocol consumers should use type-only imports where possible. Protocol must not import `ai`, `zustand`, `hono`, `node:*`, `bun:*`, React, apps, or agent-core.
- Bun binary asset strategy: use true single-binary embedding with generated `import ... with { type: "file" }` manifest and `Bun.build({ files, entrypoints, compile, naming: { asset: "[name].[ext]" } })`. Do not rely on Hono `serveStatic` for embedded assets.

## Work Objectives

### Core Objective
Create a dependency-clean Bun/Turbo monorepo while preserving Specra runtime behavior, eliminating duplicated Web/session protocol types, preventing browser code from importing agent runtime internals, and producing a standalone binary with embedded Web UI.

### Deliverables
- Root Bun workspace + Turbo configuration.
- `packages/protocol` with canonical `Session*` naming and pure reducer.
- `packages/agent-core` with runtime internals and server-facing public facade.
- `apps/server` Hono adapter and binary entry point.
- `apps/web` React/Vite app isolated from runtime internals.
- Generated embedded Web asset manifest and custom embedded static serving path.
- Updated root/package scripts.
- Rewritten architecture tests.
- Updated colocated tests/imports after moves.
- Verified `dist/specra` binary serving `/api/health`, `/`, SPA fallback route, and hashed assets without Vite or source tree.

### Definition of Done (verifiable conditions with commands)
- `bun run typecheck` exits 0 from repository root.
- `bun test` exits 0 from repository root.
- `bun run build` exits 0 and produces executable `dist/specra`.
- `./dist/specra` starts Hono server on default `4096` without Vite.
- `curl -sSI http://localhost:4096/api/health` returns HTTP 200 and `curl -sS http://localhost:4096/api/health` returns a valid JSON health payload matching the implemented health-route contract.
- `curl -sSI http://localhost:4096/` returns HTTP 200 and `content-type` containing `text/html`.
- `curl -sSI http://localhost:4096/projects/example/deep/spa/route` returns HTML SPA fallback with HTTP 200, unless that route is an API route.
- At least one hashed Vite asset referenced by embedded `index.html` returns HTTP 200 from `./dist/specra`.
- Running the binary from a temporary directory while the repository `apps/web/dist` is unavailable still serves the UI, proving true embedding.
- Architecture tests prove:
  - `apps/web` does not import `@specra/agent-core`, `@specra/server`, `node:*`, or `bun:*` from runtime source.
  - `packages/protocol` does not import `ai`, `zustand`, `hono`, `node:*`, `bun:*`, React, apps, or `@specra/agent-core`.
  - `packages/agent-core` does not import `hono`, `apps/server`, `apps/web`, React, or React DOM.
  - `apps/server` does not import `apps/web` or React.

- `@specra/agent-core` package-root exports only intentional server-facing APIs.

- Explicit pause/escalation rules for new critical architecture decisions.

### Must NOT Have

- No `packages/workflow` in this migration.
- No `apps/docs` migration in this migration.
- No `packages/typescript-config` in this migration; use root `tsconfig.base.json`.
- No generated SDK/OpenAPI/public plugin platform in this migration.
- No `apps/web -> @specra/agent-core` imports.
- No protocol imports from runtime/server/browser-framework internals.
- No Hono `serveStatic` for embedded binary assets.
- No root `start` or `server:prod` scripts after migration.
- No acceptance criteria requiring manual human UI inspection.

## Verification Strategy
> ZERO HUMAN INTERVENTION - all verification is agent-executed.
- Test decision: TDD where practical for new boundary/build behavior; tests-after for mechanical file moves; framework is existing `bun:test`.
- QA policy: Every task has agent-executed happy and failure/edge scenarios.
- Evidence: `.sisyphus/evidence/task-{N}-{slug}.{ext}`.
- Validation order follows project convention: `bun run typecheck` → `bun test`.
- For binary work, verification additionally runs compiled artifact and records curl output.
- For architecture work, verification records boundary-test output and grep-style no-import checks.

## Global Escalation Rules
Executors must pause and present evidence/options/recommendation to the user before continuing if any of these occur:
- A new circular dependency appears that is not covered by this plan.
- More than 10 new exports are proposed for `@specra/protocol` beyond planned protocol/session/API DTOs and pure reducer.
- Any implementation requires `apps/web` to import `@specra/agent-core`.
- Bun compile cannot embed or serve Web assets using the manifest approach.
- Turbo cannot preserve root command semantics.
- LSP auto-installer behaves unexpectedly in the compiled binary, such as running during startup without user-triggered LSP usage.
- Any proposed fix would introduce a new package not listed in this plan.
